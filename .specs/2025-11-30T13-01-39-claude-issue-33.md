## Overview

### Summary

Currently, the "In The Works" page (`src/app/in-the-works/page.tsx`) displays Projects and Events in a 2-column grid (on wide screens) with images cropped to fit a fixed `h-48` (192px) container using `object-cover`. The images are positioned above the text content. This implementation plan changes the layout to:

1. Display **one Project or Event per row** on wide screens (instead of 2-per-row)
2. Position the **image to the left** of the text on wide screens
3. Show the **full, uncropped image** in its original aspect ratio
4. Constrain images to **400px height on wide screens** with variable width
5. On mobile, keep images **above the text**, constrained to container width with variable height

### Goals and Success Criteria

**Goals:**

- Improve user experience by showing full, uncropped images
- Create a more spacious, gallery-like layout for Projects and Events
- Maintain responsive design principles for mobile and desktop

## Technical Approach

### Architectural Decisions and Rationale

#### 1. **CSS Grid to Flexbox Transition**

- **Decision:** Change from `grid grid-cols-1 md:grid-cols-2` to a single-column layout with flexbox for horizontal alignment on desktop
- **Rationale:** Flexbox provides better control for side-by-side layouts with variable-width images
- **SOLID Principle:** Single Responsibility - Each layout section (Projects, Events) has one clear presentation pattern

#### 2. **Responsive Image Constraints**

- **Decision:** Use Next.js `<Image>` with explicit width/height to preserve aspect ratio
- **Rationale:**
    - Setting `width` and `height` props (based on intrinsic dimensions) prevents layout shift
    - Remove `fill` and `object-cover` which crops images
    - CSS constraints (`max-h-[400px]` on desktop, `w-full` on mobile) maintain size limits
- **SOLID Principle:** Open/Closed - Image component is extended with new sizing behavior without modifying Next.js internals

#### 3. **Conditional Rendering for Desktop vs Mobile**

- **Decision:** Use Tailwind responsive classes (`md:flex-row`, `flex-col`) to switch layouts
- **Rationale:** Tailwind's mobile-first approach makes responsive design declarative and maintainable
- **SOLID Principle:** Interface Segregation - Layout concerns are separated via responsive utility classes

#### 4. **Preserve Image Aspect Ratio**

- **Decision:** Use `object-contain` instead of `object-cover` and allow container to flex
- **Rationale:** `object-contain` ensures the full image is visible without cropping
- **SOLID Principle:** Dependency Inversion - We depend on the abstraction of "preserve aspect ratio" rather than specific cropping behavior

### Design Patterns to Be Used

1. **Container/Presentational Pattern:** The page remains a presentational component with data fetching at the top
2. **Responsive Design Pattern:** Mobile-first approach with breakpoint-based layout changes
3. **Component Composition:** Separate sections for Projects and Events with consistent internal structure

### Integration Points with Existing Code

- **Database Queries:** No changes required to `getAllProjects()` or `getUpcomingEvents()`
- **Image URLs:** Continue using `image_url` field from database
- **TypeScript Types:** Existing `ProjectRow` and `EventRow` types remain unchanged
- **Styling:** Uses existing Tailwind configuration

### Potential Risks and Mitigation Strategies

| Risk                                                                              | Impact | Mitigation                                                                       |
| --------------------------------------------------------------------------------- | ------ | -------------------------------------------------------------------------------- |
| Images with extreme aspect ratios (very wide or very tall) create awkward layouts | Medium | Test with various aspect ratios; consider max-width constraints if needed        |
| Performance impact from larger images                                             | Low    | Next.js Image component handles optimization; already using `revalidate` for ISR |
| Browser compatibility for flexbox responsive behavior                             | Low    | Flexbox has excellent browser support; test in target browsers                   |
| Broken layout if image_url is missing                                             | Low    | Existing conditional rendering handles missing images                            |

### SOLID Principles Application

1. **Single Responsibility Principle (SRP)**
    - Each section (Projects, Events) has a single responsibility: display its content type
    - Layout logic is separated from data fetching logic
    - Image display logic is encapsulated within its container

2. **Open/Closed Principle (OCP)**
    - Page is open for extension (can add new sections) but closed for modification
    - Image sizing approach is extended without modifying core Image component

3. **Liskov Substitution Principle (LSP)**
    - Not directly applicable (no inheritance), but component behavior remains consistent
    - Projects and Events can be rendered interchangeably with the same layout pattern

4. **Interface Segregation Principle (ISP)**
    - Layout concerns are segregated via responsive utility classes
    - Desktop and mobile layouts are defined separately

5. **Dependency Inversion Principle (DIP)**
    - Component depends on database query abstractions (`getAllProjects`, `getUpcomingEvents`)
    - Image display depends on Next.js Image abstraction, not specific browser APIs
